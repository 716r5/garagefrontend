# Garage@EEE Website - Technical Overview

## Project Summary
React-based frontend for Garage@EEE's website (https://garage-eee.com/). Uses Google Sheets as a backend database accessed through Google Apps Script APIs. Built with Create React App, deployed via GitHub Pages.

## Core Architecture

### Tech Stack
- **Framework**: React 18.1.0
- **Router**: React Router DOM v6.21.1 (with hash link support)
- **Styling**: CSS Modules (component-level scoping)
- **Animations**: Framer Motion v11.0.14
- **Smooth Scroll**: Lenis v1.0.45
- **HTTP Client**: Axios v1.8.4
- **Build Tool**: react-scripts v5.0.1
- **Deployment**: gh-pages package

### Data Flow Pattern
All data comes from Google Sheets → Apps Script API endpoints → Frontend fetch → Component render. No traditional backend server (means you don't have to build/maintain your own backend server - google is handling that part for you). Authentication uses JWT tokens generated by Apps Script.

## Contexts (`src/contexts/`)

**AuthProvider.js**
- React Context for authentication state
- Stores: name, matric, passcode, token
- Uses localStorage for persistence (authToken, name, matric)
- loginAction(responseData): receives JWT and user info, sets state
- logoutAction(wasExpired): clears auth state, navigates home, reloads page, clears cart
- Tracks authStatus (expired/loggedOut) for session timeout warnings
- useAuth() hook for consuming context

**CartProvider.js**
- Shopping cart state management
- Stores: userCredits, cartCount, cartItems (array of {itemName, quantity, unitPrice, image})
- Persists to localStorage
- setCredits(credits): updates user's credit balance
- setCart(count, items): updates cart contents
- clearCart(): wipes cart and credits (used on logout)
- useCart() hook for consuming context

---

## Hooks (`src/hooks/`)

**useFetch.js**
- Custom hook for API calls with caching
- Parameters: url, headers (optional), enabled (default true), useCache (default true)
- Handles AbortController for cancelling requests
- 5-minute cache expiration (localStorage)
- Special handling for shopInventory: checks cacheVersion on server before using cache
- Token validation: if "Invalid token" error, calls logoutAction(true) to force re-login
- Returns: { data, isLoading, error }

**useFetchPoints.js**
- Specialized hook for fetching user's Innocredit balance
- Requires matric number and token
- Posts to API with type="shopData"
- Updates CartProvider with credits on successful fetch
- Handles token expiration
- Returns: { userCredits, loading }

**useBreakpoint.js**
- Window resize listener
- Returns: "desktop" (>=1200px), "tablet" (>=640px), or "mobile" (<640px)
- Used throughout components for responsive behavior

## Key Patterns & Architecture Notes

### State Management
- **Global State**: Context API (AuthProvider, CartProvider)
- **Local State**: useState in individual components
- **Persistence**: localStorage for auth token, cart data, cache
- **No Redux**: Simple enough for Context API

### Data Fetching
- All via custom useFetch hook
- No GraphQL, just REST-style GET/POST to Apps Script
- Query params in URL: `?type=X&fields=Y&index=Z&token=ABC`
- Caching implemented at hook level (not using React Query/SWR)

### Routing Strategy
- Client-side routing with React Router v6
- Hash links for in-page navigation (/#section)
- Protected routes via PrivateRoute HOC pattern
- GitHub Pages compatibility script in index.html

### Styling Approach
- CSS Modules for component isolation
- Global variables in index.css for consistency
- No CSS-in-JS libraries
- Responsive breakpoints: 1200px, 640px

### Animation Libraries
- Framer Motion for page transitions, modals, menu animations
- Lenis for smooth scroll (better than CSS scroll-behavior)
- AOS library included but usage unclear

### Authentication Flow
1. User attempts protected route → PrivateRoute redirects to /login
2. Login form posts matric + passcode to LOGIN_DOMAIN
3. Apps Script validates against Google Sheets DB
4. Returns JWT token + user data
5. Frontend stores in AuthProvider context + localStorage
6. Token included in subsequent API calls for protected data
7. Invalid token triggers automatic logout

### Shop/Credits System
- "Innocredits" are internal currency
- User balance fetched from Google Sheets via matric lookup
- Cart managed in CartProvider, synced to localStorage
- Purchase flow: Shop → Checkout → Acknowledgement
- Transaction recorded in Google Sheets via PURCHASE_API_DOMAIN

### File Upload System
1. User selects files in FileUploader (drag-drop or dialog)
2. Files encoded to base64 in ContactUs component
3. POST to CONTACT_US_UPLOAD_DOMAIN with JSON payload
4. Apps Script decodes base64, saves to Google Drive
5. Form data logged in Google Sheets
6. Confirmation emails sent to user + admin

## Known Issues & Technical Debt

From TinkeringProjDocumentation.md:
1. **Responsiveness**: Tinkering page layout breaks on some tablet dimensions
2. **Hash link scrolling**: Smooth scroll sometimes misfires when navigating from different page (page not fully loaded when scroll command fires)
3. **Hard-coded layouts**: Some image grids have fixed layouts that don't adapt well

From general code inspection:
- **No TypeScript**: tsconfig.json present but code is JavaScript (inconsistency)
- **Mixed file extensions**: Some components use .js, others .jsx (no clear convention)
- **Global CSS class names**: Risk of collisions despite CSS Modules
- **No error boundaries**: App could crash entirely on component error
- **Minimal test coverage**: Test files exist but likely minimal
- **No CI/CD visible**: Deployment via manual `npm run deploy` command
- **Cache invalidation**: useFetch cache only checks version for shopInventory, not other data types
- **Token expiration**: No automatic token refresh, user must re-login
- **Accessibility**: No mention of ARIA labels, keyboard nav testing, screen reader support

## Backend Architecture (External)

While this is a frontend repo, understanding the backend is impt!:
- **Database**: Google Sheets (no SQL database)
- **API Layer**: Google Apps Script (JavaScript runtime)
- **File Storage**: Google Drive folders
- **Email**: Apps Script MailApp service
- **Authentication**: Custom JWT implementation in Apps Script
- **Limitations**: Apps Script has execution time limits (6 min), quota limits (email sends, drive operations)

Google Sheets structure:
- Different sheets for: home content, projects, ambassadors, events, facilities, newsletters, members (auth), shop inventory, assigned projects, tinkering data
- Each sheet has expected column structure (documented in APIDocumentation.md)
- Apps Script reads sheets, processes data, returns JSON

## Revamp Considerations

### Immediate Issues
1. **Fix hash link scrolling**: Add loading state check before scrollTo, or delay scroll command
2. **Standardize file extensions**: Choose .jsx for React components or .js with clear convention
3. **Improve tablet responsiveness**: Review TinkeringProject.module.css breakpoints
4. **Add error boundaries**: Wrap major route sections to prevent full app crashes
5. **Accessibility audit**: Add ARIA labels, test keyboard navigation, ensure color contrast

### Architecture Improvements
1. **TypeScript migration**: tsconfig.json is there, commit to it or remove it
2. **Better state management**: Consider Zustand or Jotai for simpler global state (Context can get messy)
3. **API layer abstraction**: Create services folder with typed API calls instead of scattered useFetch calls
4. **Component library decision**: Either commit to building custom components or adopt shadcn/ui, Mantine, etc.
5. **Form validation library**: Add React Hook Form + Zod for ContactUs and Login forms
6. **Testing strategy**: Add Vitest, React Testing Library tests for critical paths (auth, shop flow)

### Performance Optimizations
1. **Image optimization**: Serve images through CDN, use modern formats (WebP, AVIF)
2. **Code splitting**: Lazy load routes, reduce initial bundle size
3. **useFetch improvements**: Implement proper stale-while-revalidate caching strategy (SWR pattern)
4. **Remove unused dependencies**: aos appears unused, @react-google-maps/api might be redundant

### UX Enhancements
1. **Loading skeletons**: Replace LoadingSpinner with skeleton screens for better perceived performance
2. **Error states**: Show friendly error messages when API calls fail, not just console.log
3. **Optimistic updates**: Update UI immediately for cart actions, rollback if API fails
4. **Form persistence**: Save form progress to localStorage to prevent data loss
5. **Session timeout warning**: Show warning before token expires, offer extension

### Backend Migration Path
If you ever want to move off Google Sheets:
1. Keep API endpoint structure same (minimize frontend changes)
2. Replace Apps Script with Node.js/Express or Next.js API routes
3. Migrate Google Sheets to PostgreSQL or MongoDB
4. Move file uploads to S3/Cloudflare R2
5. Use proper auth service (Auth0, Clerk, Supabase)

### Scalability Concerns
- Google Sheets has row limits (10 million cells per sheet, but performance degrades way before)
- Apps Script quotas could be hit with high traffic
- No CDN for images (all served from Google Drive/Sheets)
- No server-side rendering (all client-side, slower initial load)

### Security Notes
- JWT tokens stored in localStorage (vulnerable to XSS, but acceptable for this use case)
- No HTTPS enforcement visible in code (relies on GitHub Pages default)
- API keys in .env not in repo (good), but Apps Script URLs are public (anyone can call if they know the URL)
- No rate limiting visible (Apps Script might have some, but not controlled by frontend)
